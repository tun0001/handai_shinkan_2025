## 1. はじめに

このマニュアルは、大阪大学陸上競技部の新歓サイトを引き継ぎ、運用するためのガイドです。プログラミングの経験がない方でも、このマニュアルに従えば、サイトの内容を更新し、公開することができます。

### このサイトでできること

- 新歓イベント情報の更新
- 主将やパート長のメッセージ変更
- メンバー紹介の更新
- よくある質問（Q&A）の編集
- 画像の差し替え
- SNSリンクの更新

### 必要な作業時間の目安

- **初回セットアップ**: 30分〜1時間
- **文章の編集**: 10〜30分
- **画像の差し替え**: 5〜15分
- **サイトの公開**: 5〜10分

## 3. 引き継ぎの準備（リポジトリのFork）

### 3.1 前提条件

このプロジェクトは**公開リポジトリ（Public Repository）**として管理されています。引き継ぎ者は以下の手順でリポジトリをFork（複製）して、自分のGitHubアカウントで管理・公開します。

### 3.2 GitHubアカウントの準備

#### 個人アカウントを使う場合

1. [GitHub](https://github.com/)でアカウントを作成（既にある場合はスキップ）
2. アカウント名は個人用でOK

#### HP係用の共有アカウントを作る場合（推奨）

後述の「[11. 今後の運用について](#11-今後の運用についてhp係用アカウントの推奨)」を参照してください。共有アカウントを使うことで、引き継ぎがよりスムーズになります。

### 3.3 リポジトリのFork手順

#### ステップ1: 元のリポジトリにアクセス

1. ブラウザで以下のURLにアクセス：
   ```
   https://github.com/tun0001/handai_shinkan_2025
   ```

2. ページ右上の **「Fork」** ボタンをクリック

#### ステップ2: Forkの作成

1. Fork先のアカウントを選択（自分のアカウント）
2. リポジトリ名は **そのまま `handai_shinkan_2025`** を推奨
   - または年度に合わせて `handai_shinkan_2026` などに変更してもOK
3. **「Create fork」** をクリック

数秒で、自分のアカウントにリポジトリがコピーされます。

#### ステップ3: Forkしたリポジトリの確認

Fork完了後、URLが以下のように変わります：
```
https://github.com/【あなたのアカウント名】/handai_shinkan_2025
```

これが今後編集・管理するリポジトリになります。

### 3.4 GitHub Pagesの設定

ForkしたリポジトリをGitHub Pagesで公開するための設定を行います。

#### ステップ1: Settingsを開く

1. Forkしたリポジトリのページで、上部メニューの **「Settings」** をクリック
2. 左サイドバーから **「Pages」** をクリック

#### ステップ2: Sourceの設定

**「Build and deployment」** セクションで以下を設定：

1. **Source**: `Deploy from a branch` を選択
2. **Branch**: 
   - ブランチ: `main` を選択
   - フォルダ: `/docs` を選択
3. **「Save」** をクリック

#### ステップ3: 公開URLの確認

設定後、数分待つと以下のような公開URLが表示されます：
```
https://【あなたのアカウント名】.github.io/handai_shinkan_2025/
```

このURLが新しい新歓サイトのURLになります。

⚠️ **注意**: 初回は `docs/` フォルダがまだ存在しないため、一度 `npm run generate` と `npm run deploy` を実行する必要があります（後述）。`npm run generate` で `docs/` フォルダに静的ファイルが生成され、それをGitにプッシュすることで公開されます。

#### ステップ4: カスタムドメイン（任意）

独自ドメインを使いたい場合は、**「Custom domain」** にドメイン名を入力できます（上級者向け）。

### 3.5 nuxt.config.jsの修正

⚠️ **重要**: リポジトリ名を変更した場合、設定ファイルも修正が必要です。

**ファイル**: `nuxt.config.js`

```javascript
const routerBase =
  process.env.DEPLOY_ENV === 'GH_PAGES'
    ? {
      router: {
        base: '/handai_shinkan_2025/',  // ← リポジトリ名と一致させる
      },
    }
    : {}
```

**例**: リポジトリ名を `handai_shinkan_2026` に変更した場合
```javascript
base: '/handai_shinkan_2026/',  // ← 新しいリポジトリ名
```

### 3.6 Forkした後の最初のデプロイ

ForkとGitHub Pagesの設定が完了したら、ローカルで作業を開始します。

次のセクション「[4. 環境セットアップ](#4-環境セットアップ)」に進んでください。

## 4. 環境セットアップ

### 3.1 必要なソフトウェアのインストール

#### ① Node.jsのインストール

Node.jsは、このサイトを動かすために必要なソフトウェアです。

1. [Node.js公式サイト](https://nodejs.org/)にアクセス
2. **LTS版**（推奨版）をダウンロード
3. ダウンロードしたファイルを開いてインストール
4. インストール完了後、確認のためにターミナル（コマンドプロンプト）を開いて以下を入力：
   ```bash
   node --version
   ```
   バージョン番号（例: v18.17.0）が表示されればOK

#### ② Gitのインストール（既にインストールされている場合はスキップ）

Gitは、プログラムのバージョン管理に使います。

1. [Git公式サイト](https://git-scm.com/)からダウンロード
2. インストール（基本的にデフォルト設定でOK）
3. 確認：
   ```bash
   git --version
   ```

#### ③ VSCode（エディタ）のインストール（推奨）

VSCodeは、コードを編集するためのソフトです。

1. [VSCode公式サイト](https://code.visualstudio.com/)からダウンロード
2. インストール
3. 推奨拡張機能のインストール（VSCodeを開いてから）：
   - **Vetur**: Vue.jsファイルの編集に便利
   - **Prettier**: コードを自動で整形
   - **Japanese Language Pack**: 日本語化

### 3.2 プロジェクトのセットアップ

#### ① プロジェクトのダウンロード

ターミナル（またはVSCode内のターミナル）を開いて、以下を実行：

```bash
# プロジェクトをダウンロード（Forkした自分のリポジトリから）
git clone https://github.com/【あなたのアカウント名】/handai_shinkan_2025.git

# 例: HP係用アカウント「outf-hp」の場合
# git clone https://github.com/outf-hp/handai_shinkan_2025.git

# プロジェクトフォルダに移動
cd handai_shinkan_2025

# 必要なパッケージをインストール（初回のみ）
npm install
```

⚠️ `npm install` は初回のみ実行します。数分かかることがあります。

#### ② VSCodeでプロジェクトを開く

VSCodeを起動して、「ファイル」→「フォルダーを開く」から `handai_shinkan_2025` フォルダを選択します。

## 6. 文章の編集方法

### 5.1 基本的な編集の流れ

1. VSCodeで該当ファイルを開く
2. 文章を探して編集
3. 保存（`Ctrl + S` または `Cmd + S`）
4. `npm run dev` でプレビュー確認
5. 問題なければ公開作業へ

### 5.2 JavaScriptファイルの文章編集のポイント

#### 文字列の書き方

文章は **シングルクォート `'`** または **バッククォート `` ` ``** で囲みます。

```javascript
// ✅ 正しい例
title: '大阪大学陸上部',
message: 'こんにちは！',

// ❌ 間違った例
title: 大阪大学陸上部,  // クォートがない
message: "こんにちは！",  // ダブルクォートは避ける
```

#### 改行の書き方

改行したい場合は `\n` を使います。

```javascript
txt: '1行目\n2行目\n3行目',
// ↓ 実際の表示
// 1行目
// 2行目
// 3行目
```

#### 長い文章の書き方

長い文章は **バッククォート** を使うと便利です。

```javascript
// 方法1: シングルクォート + \n
txt: '文章1\n文章2\n文章3',

// 方法2: バッククォート（複数行OK）
txt: `文章1
文章2
文章3`,
```

#### 特殊文字に注意

以下の文字は特別な意味を持つため、そのまま使えません：

- シングルクォート `'` → `\'` とエスケープ
- バックスラッシュ `\` → `\\` と2つ書く

```javascript
// 例: アポストロフィを使いたい場合
txt: 'It\'s a nice day!',  // ✅ 正しい
txt: 'It's a nice day!',   // ❌ エラーになる
```

## 7. 画像の差し替え方法

### 6.1 画像の準備

#### 推奨される画像形式
- **形式**: JPG または PNG
- **サイズ**: 用途による（下記参照）
- **ファイル名**: 英数字とハイフンのみ（日本語は避ける）

#### 各セクションの推奨画像サイズ

| 用途 | 推奨サイズ | 配置場所 |
|-----|----------|---------|
| パート長の顔写真 | 400×400px 程度 | `assets/member-img/` |
| イベント画像 | 800×600px 程度 | `assets/shinkanjoho/Img/` |
| メッセージ背景 | 1920×1080px 程度 | `assets/message/img/` |
| 活動の流れ画像 | 800×600px 程度 | `assets/activity/` |

### 6.2 画像の配置手順

#### 手順1: 画像ファイルの配置

1. 差し替える画像を用意
2. VSCodeで該当フォルダを開く
3. 画像ファイルをフォルダにドラッグ&ドロップ

**例**: パート長の写真を追加する場合
```
assets/member-img/ に「フィールド-2026.jpg」を配置
```

#### 手順2: ファイル名の確認

ファイル名に以下が含まれていないか確認：
- 日本語
- スペース
- 特殊文字（`@`, `#`, `%` など）

**良い例**: `field-2026.jpg`, `member-01.png`  
**悪い例**: `フィールド 写真.jpg`, `member@01.png`

#### 手順3: コード内で画像を参照

該当するJavaScriptファイル（`.js`）を開いて、画像のパスを更新します。

```javascript
// 例: sections/MemberSec/script.js
{
  name: '嘉藤和真',
  txt: 'フィールドパート長',
  imgSrc: require('@/assets/member-img/フィールド-2025.jpg'),  // ← ここを編集
}

// ↓ 新しい画像に変更

{
  name: '山田太郎',  // 名前も変更
  txt: 'フィールドパート長',
  imgSrc: require('@/assets/member-img/フィールド-2026.jpg'),  // ← 新しいファイル名
}
```

### 6.3 画像パスの書き方

```javascript
// 基本形式
imgSrc: require('@/assets/フォルダ名/ファイル名.jpg'),

// @/ は「プロジェクトのルート」を表す
// require() は画像を読み込むための関数
```

```javascript
          detail: '部活動に関することは...',                     // 詳細説明
          dayandtime: '日時：3月15日(土) 4月1日(火)',           // 日時（詳細表示用）
          place: 'zoom参加はこちらから↓',                       // 場所
          zoomlink: 'https://...',                              // ZoomリンクやフォームURL
          zoomID: 'ミーティングID: 838 2306 5119',             // Zoom ID（任意）
          zoompass: 'パスコード: 564534',                       // Zoomパスコード（任意）

        // 2つ目のイベント
        {
          imgSrc: require('@/assets/shinkanjoho/Img/img2.jpg'),
          date: '4月5日(土),4月6日(日)',
          title: 'サーオリ',
          // ...
        },
      ],
    }
  },
}
```

#### イベントの追加方法

配列の最後に新しいオブジェクトを追加します：

```javascript
events: [
  // 既存のイベント...
  {
    imgSrc: require('@/assets/shinkanjoho/Img/img6.jpg'),
    date: '5月10日(土)',
    title: '練習体験会',
    detail: '実際にすいらんで練習体験ができます！',
    dayandtime: '日時：5月10日(土) 14:00〜',
    place: '場所：すいらん',
  },
],
```

#### イベントの削除方法

該当する `{ }` ブロック全体をコメントアウトまたは削除します：

```javascript
events: [
  // {
  //   imgSrc: require('@/assets/shinkanjoho/Img/img3.jpg'),
  //   date: '日時未定',
  //   title: '女子新歓',
  //   // ...
  // },  ← このイベントはコメントアウトされて非表示
],
```

## 11. 今後の運用について（HP係用アカウントの推奨）

### 11.1 なぜHP係用アカウントが必要か

現在、個人のGitHubアカウントでリポジトリを管理している場合、引き継ぎ時に以下の問題が発生する可能性があります：

#### 問題点

1. **毎年Forkが必要**
   - 引継ぎの度に新しいアカウントでForkし直す必要がある
   - 過去の編集履歴が引き継がれない

2. **URLが変わる**
   - アカウント名が変わるため、公開URLも変わる
   - 例: `https://tanaka.github.io/...` → `https://suzuki.github.io/...`
   - SNSやポスターに記載したURLが使えなくなる

3. **アクセス管理が難しい**
   - 個人アカウントでは、複数人での共同編集が困難
   - 前任者がアカウントを削除すると、リポジトリも消える可能性

### 11.2 HP係用アカウントのメリット

**部活動専用の共有GitHubアカウント**を作成すると、以下のメリットがあります：

#### ✅ 継続性
- アカウント名が変わらないため、**URLが永続的に使える**
- 例: `https://outf-hp.github.io/shinkan/`（毎年同じURL）

#### ✅ 引き継ぎが簡単
- Forkし直す必要がない
- パスワードやアクセストークンを引き継ぐだけ
- 過去の編集履歴もすべて残る

#### ✅ 複数人での管理
- 必要に応じて、複数人に編集権限を付与できる
- GitHubの「Collaborators」機能を使用

#### ✅ ブランディング
- 部活動の公式アカウントとして認識される
- 他のリポジトリ（公式HP、部誌など）も一元管理できる

### 11.3 HP係用アカウントの作成手順

#### ステップ1: 新しいGitHubアカウントを作成

1. [GitHub](https://github.com/)にアクセス
2. 「Sign up」をクリック
3. 以下の情報を入力：
   - **Username**: 部活動を表す名前（例: `outf-hp`, `handai-tf-web`）
   - **Email**: 部活動の共有メールアドレス、またはHP係で引き継げるアドレス
   - **Password**: 安全なパスワード（引継ぎ資料に記録）

#### ステップ2: メールアドレスの認証

GitHubから送られてくる認証メールを確認して、アカウントを有効化します。

#### ステップ3: リポジトリの移管またはFork

**方法A: 既存リポジトリの移管（推奨）**

既に個人アカウントでForkしている場合：

1. リポジトリの「Settings」→「Transfer ownership」
2. 新しいアカウント名を入力して移管
3. これにより、編集履歴もすべて引き継がれる

**方法B: 新しくFork**

元のリポジトリ（`tun0001/handai_shinkan_2025`）から、新しいアカウントでForkします。

#### ステップ4: GitHub Pagesの再設定

新しいアカウントで「[3.4 GitHub Pagesの設定](#34-github-pagesの設定)」を実施します。

### 11.4 アカウント情報の管理

#### セキュリティ

- **パスワードは安全に管理**
  - パスワード管理ツール（1Password、Bitwardenなど）の使用を推奨
  - 引継ぎ資料に記載する場合は、紙媒体で施錠できる場所に保管

- **二段階認証（2FA）の設定**
  - GitHubの「Settings」→「Password and authentication」
  - 認証アプリ（Google Authenticator、Authyなど）を使用
  - バックアップコードを印刷して保管

#### 引き継ぎ資料に記載すべき情報

```
【HP係用GitHubアカウント情報】
- アカウント名: outf-hp
- メールアドレス: ********@gmail.com
- パスワード: （安全な場所に保管）
- 2FA: （バックアップコード別紙）
- リポジトリURL: https://github.com/outf-hp/handai_shinkan_2025
- 公開URL: https://outf-hp.github.io/handai_shinkan_2025/
```

### 11.5 今後の運用フロー（HP係用アカウント使用時）

#### 年度更新時

1. 前任者から**アカウント情報**を引き継ぐ
2. 自分のPCにリポジトリをクローン
   ```bash
   git clone https://github.com/outf-hp/handai_shinkan_2025.git
   ```
3. 内容を更新（イベント情報、メンバー情報など）
4. GitHub Pagesに公開
   ```bash

   ```
5. **同じURLで新しい内容が公開される**

#### 複数人で編集する場合

1. リポジトリの「Settings」→「Collaborators」
2. 編集者のGitHubアカウントを追加
3. 各自が自分のアカウントで編集できる

## 12. トラブルシューティング

### 12.1 インストール・環境関連

#### 問題1: `npm install` でエラーが出る

**症状**: パッケージのインストールに失敗する

**解決策**:

```bash
# キャッシュをクリア
npm cache clean --force

npm install
```

#### 問題2: `npm run dev` でエラーが出る

**症状**: 開発サーバーが起動しない

**解決策**:
```bash
# 環境変数を設定してから実行
export NODE_OPTIONS=--openssl-legacy-provider
npm run dev
```

Windowsの場合：
```cmd
set NODE_OPTIONS=--openssl-legacy-provider
npm run dev
```

### 12.2 表示・ファイル関連

#### 問題3: 画像が表示されない

**症状**: サイトで画像が表示されず、404エラーが出る

**原因と解決策**:

#### 原因1: ファイル名が間違っている
```javascript
// ❌ 間違い
imgSrc: require('@/assets/member-img/フィールド-2026.jpg'),
// 実際のファイル名: field-2026.jpg

// ✅ 正しい
imgSrc: require('@/assets/member-img/field-2026.jpg'),
```

#### 原因2: ファイルパスが間違っている
```javascript
// ❌ 間違い
imgSrc: require('@/assets/img/member-img/field-2026.jpg'),

// ✅ 正しい
imgSrc: require('@/assets/member-img/field-2026.jpg'),
```

#### 原因3: ファイルが配置されていない
- フォルダに画像ファイルが実際に存在するか確認

#### 問題4: 編集が反映されない

**解決策**:

1. **ファイルを保存したか確認**
   - `Ctrl + S` (Windows) または `Cmd + S` (Mac)

2. **開発サーバーを再起動**
   ```bash
   # Ctrl + C でサーバーを停止
   # 再度起動
   npm run dev
   ```

3. **ブラウザのキャッシュをクリア**
   - `Ctrl + Shift + R` (Windows) または `Cmd + Shift + R` (Mac)

#### 問題5: 構文エラーが出る

**症状**: JavaScriptの構文エラーでサイトが表示されない

**よくあるミス**:

#### ミス1: カンマ忘れ
```javascript
// ❌ エラー
{
  name: '田中太郎'
  txt: 'パート長'  // ← カンマがない
}

// ✅ 正しい
{
  name: '田中太郎',
  txt: 'パート長',  // ← カンマを追加
}
```

#### ミス2: クォート忘れ
```javascript
// ❌ エラー
name: 田中太郎,

// ✅ 正しい
name: '田中太郎',
```

#### ミス3: 括弧の対応が合わない
```javascript
// ❌ エラー
events: [
  { title: 'イベント1' },
  { title: 'イベント2' },
]  // ← 対応する [ がある

// 確認方法: VSCodeで括弧の色やインデントを確認
```

**解決策**:
- VSCodeでエラー箇所を確認（赤い波線が表示される）
- エラーメッセージを読む（行番号が表示される）

### 12.3 デプロイ関連

#### 問題6: デプロイに失敗する

**症状**: `npm run deploy` でエラーが出る

**解決策**:

1. **Gitの設定を確認**
   ```bash
   git config user.name
   git config user.email
   ```

2. **GitHubの認証を確認**
   ```bash
   # SSHの場合
   ssh -T git@github.com
   
   # HTTPSの場合
   git remote -v
   ```

3. **generate後にdeployを実行**
   ```bash
   npm run generate
   npm run deploy
   ```

#### 問題7: GitHub Pagesで404エラーが出る

**症状**: サイトにアクセスすると「404 Not Found」が表示される

**原因と解決策**:

1. **docs/フォルダが存在しない、または空**
   ```bash
   # 一度generateとdeployを実行
   npm run generate
   npm run deploy
   ```

2. **GitHub Pagesの設定が間違っている**
   - リポジトリの「Settings」→「Pages」を確認
   - Source: `main` ブランチ、`/docs` フォルダに設定

3. **nuxt.config.jsのbaseパスが間違っている**
   ```javascript
   // リポジトリ名と一致しているか確認
   router: {
     base: '/handai_shinkan_2025/',  // ← リポジトリ名
   }
   ```

4. **デプロイ後、反映に時間がかかる**
   - 初回は5〜10分待つ
   - ブラウザのキャッシュをクリア（Ctrl + Shift + R）

#### 問題8: Forkしたのにリポジトリが見つからない

**解決策**:

1. GitHubのプロフィールページから「Repositories」タブを確認
2. リポジトリ名で検索
3. Fork元のリポジトリページで「Forks」を確認

## 13. よくある質問

### Q1: プログラミングの知識がないのですが、大丈夫ですか？

**A**: 大丈夫です。このマニュアルに従えば、基本的な編集はできます。文章の変更や画像の差し替えなら、特別な知識は不要です。不明点があれば、前任者や詳しい人に聞きましょう。

### Q2: どのくらいの頻度で更新が必要ですか？

**A**: 以下のタイミングで更新します：
- **年度更新時**（3月頃）: パート長情報、主将メッセージ、年度表記など
- **新歓イベント前**（3〜4月）: イベント情報、ZoomリンクなどA
- **随時**: Q&Aの追加、写真の差し替えなど

### Q3: 複数人で作業する場合の注意点は？

**A**: 
- 同時に同じファイルを編集しないこと
- 作業前に最新版を取得：`git pull`
- 編集後は必ずGitにプッシュして共有
- 誰がどのセクションを担当するか明確にする
- HP係用アカウントを使っている場合は、Collaboratorsとして追加するとスムーズ

### Q4: 編集内容を間違えてしまいました。元に戻せますか？

**A**: Gitを使えば戻せます：

```bash
# まだコミットしていない場合
git checkout -- ファイル名

# コミット済みの場合（直前のコミットに戻す）
git reset --hard HEAD^

# 特定のコミットに戻す
git log  # コミット履歴を確認
git reset --hard コミットID
```

⚠️ `git reset --hard` は変更が失われるので注意してください。

### Q5: サイトの表示が崩れてしまいました

**A**: 
1. 最近編集したファイルを確認
2. 構文エラーがないかチェック（VSCodeの赤い波線）
3. ブラウザの開発者ツールでエラーを確認（F12キー）
4. 元に戻す（前述のGitコマンド）

### Q6: 新しいセクションを追加したいです

**A**: 新しいセクションの追加は、既存のセクションをコピーして改造するのが簡単です。ただし、Vueコンポーネントの知識が必要になるため、詳しい人に相談することをお勧めします。

### Q7: スマートフォンでの表示を確認したいです

**A**: 
1. 開発サーバーを起動（`npm run dev`）
2. ブラウザの開発者ツールを開く（F12）
3. デバイスツールバーを表示（Ctrl + Shift + M）
4. 様々なデバイスサイズをシミュレート

または、同じWi-Fi上のスマートフォンから、PCのIPアドレスでアクセスすることも可能です。

### Q8: エラーメッセージの意味がわかりません

**A**: よくあるエラーメッセージ：

| エラーメッセージ | 意味 | 解決方法 |
|---------------|-----|---------|
| `Unexpected token` | 構文エラー | カンマやクォート、括弧を確認 |
| `Cannot find module` | ファイルが見つからない | ファイルパスや画像パスを確認 |
| `SyntaxError` | 文法エラー | コードの書き方を確認 |
| `ENOENT` | ファイルやフォルダが存在しない | パスを確認 |

詳しくは、エラーメッセージをコピーして検索すると解決策が見つかることが多いです。

### Q9: 他の大学のサークルでも使えますか？

**A**: はい、使えます。以下を変更してください：
- サークル名、ロゴ
- SNSリンク
- メンバー情報
- イベント情報
- Github PagesのURL設定（`nuxt.config.js` の `router.base`）

### Q10: このサイトを参考に新しいサイトを作りたいです

**A**: このプロジェクトはテンプレートとして利用できます：
1. プロジェクトをフォーク（コピー）
2. リポジトリ名を変更
3. `nuxt.config.js` の `router.base` を新しいリポジトリ名に変更
4. 内容をカスタマイズ

### Q11: Forkと単なるコピーの違いは？

**A**: 
- **Fork**: GitHubの機能。元のリポジトリとのつながりを保ちながら、自分のアカウントにコピー
  - 元のリポジトリの更新を取り込むことができる
  - Pull Requestで改善を提案できる
- **単なるコピー**: ファイルをダウンロードして、新しいリポジトリとしてアップロード
  - 元のリポジトリとのつながりがない

引き継ぎの場合は**Fork**を使うことを推奨します。

### Q12: 元のリポジトリ（tun0001）が削除されたらどうなりますか？

**A**: Forkしたリポジトリは独立しているため、**元のリポジトリが削除されても影響ありません**。Forkしたリポジトリは完全なコピーなので、引き続き使用できます。

### Q13: HP係用アカウントを作るタイミングは？

**A**: できるだけ早い方が良いですが、以下のタイミングがおすすめ：
- **今すぐ**: 次の引き継ぎから楽になる
- **次の年度更新時**: 引き継ぎのついでに作成
- **引継ぎが大変だと感じたとき**: 問題を感じたら即座に作成

一度作れば、以降の引き継ぎが大幅に楽になります。
